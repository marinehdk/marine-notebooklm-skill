# nlm Skill V0.1 端到端可用性评估报告

**测试日期**：2026-04-22
**测试人**：Claude (自动化 E2E)
**测试方式**：模拟真实用户全链路（setup → ask → plan → research），真实笔记本 `Claude Code实战`（287 sources）。
**覆盖命令**：`setup`、`ask`、`plan`、`research`（read-only + add-sources）

---

## 一句话结论

> 基础查询（`ask`）质量出色、可上线；`plan` 与 `research` 存在 **3 个 P0 bug**，V0.1 发布前必须修复，否则功能承诺兑现不了。

## 详细发现

### ✅ Step 1-2：`setup`（A 级）

- `setup --auth`：1s 返回 `{"authenticated": true}`，干净利落。
- `setup --project-path .`：<1s 返回完整绑定信息（notebook id/title/source_count/global_notebooks）。
- **无 UX 问题**。可直接进入 V0.1。

### 🟡 Step 3：`ask`（B+ 级，有优化空间）

三个测试 query：

| 用例 | 问题 | 置信度 | 耗时 | 质量 |
|------|------|--------|------|------|
| Q1 易 | "Claude Code hooks 是什么？" | high | 60s | ⭐⭐⭐⭐⭐ 分层结构，20+引用，覆盖全面 |
| Q2 中 | "如何配置 MCP server？" | high | 68s | ⭐⭐⭐⭐⭐ 三种配置方式+字段说明+完整示例 |
| Q3 off-topic | "K8s Ingress 选型" | low | 133s | ⭐⭐⭐⭐ 正确识别不相关，给出 research 后续建议 |

**问题**：

1. **P1 — 进度条误导**：输出 `⏳ [1/2] Querying local notebook...` 然后 `✅ [1/2]`，**永远看不到 `[2/2]`**。分母 `/2` 暗示还有下一步，但实际已结束。建议：
   - 置信度 high 时：直接写 `✅ Done (local, high confidence)`
   - 降级到 global 时：`⏳ [2/2] Escalating to global...` 才出现
2. **P2 — off-topic 查询耗时 2×**：Q3 在 `global_notebooks: []` 情况下仍 `auto_researched: true`，但耗时 133s 远超 Q1/Q2。怀疑空 global 也跑了一轮"空调用"。应短路：`if not global_notebooks: skip escalation`。

### 🚨 Step 4：`plan`（D 级，P0）

**测试**：`plan --question "Hooks vs Slash Commands 做代码风格自动化"`

**输出摘录**：
```
✅ [3/4] Scoring complete — 0/8 scored successfully
"composite_scores": { "Hooks": null, "Slash Commands": null },
"matrix": { ... 所有 cell 都是 score: null, reasoning: "", parse_warning: true ... }
```

**但是** `rationale` 字段生成了高质量文案，`recommendation: "Hooks"` 也是对的。

**症状**：
- Phase 3 scoring 100% 失败
- criterion 名字出现 `**`（markdown 粗体残留，如 `"执行的确定性与强制性 (Determinism vs. Probability)**"`）
- 最终仍给出 recommendation（fallback 路径）

**根因定位**（`scripts/lib/` 中 plan 评估器）：
- 最近 commit `ea13d1e fix(plan): batch Phase 3 score parsing` 声称修复，实际未覆盖该回归
- criterion key 里的 `**` 说明 criterion 生成阶段把 prompt 的 markdown 标记泄漏进结构化字段
- "0/8 scored" 说明批量打分 prompt 的响应解析器完全没对上

**P0 修复要点**：
1. criterion 生成阶段 strip markdown（`**X**` → `X`）
2. Phase 3 score parser 加单元测试（3-5 个模拟 LLM 响应样本）
3. 当 scoring 全失败时，**recommendation 应当降级或警告**，而不是假装成功

**延迟**：264s（4.4 分钟）— 即使修复也偏长，可考虑并行化。

### 🚨 Step 5：`research --no-add-sources`（F 级，P0）

**测试**：`research --topic "Claude Code subagent 的设计模式与适用场景"`

**灾难性结果**：返回 10 个源 **全是 Vue vs React 文章**（Wikipedia、Reddit、groovyweb、reactuse…），report 文案也是 Vue 相关：
```
"此资源列表重点分析 Vue 的核心优势、生态系统及与 React 的深度对比…"
```

所有 10 个源的 `research_task_id` 相同 → **返回了上一次研究任务的缓存结果**，没有执行新研究。5s 完成也印证这点。

**根因**（`scripts/lib/client.py:145-179`）：
```python
while time.time() < deadline:
    poll = await client.research.poll(notebook_id)
    if poll.get("status") == "completed":  # ← BUG: 不校验 task_id
        return { ... poll.get("sources") ... }
```

`client.research.poll(notebook_id)` 是 notebook-level 轮询，任何已完成的研究都会立即返回 `status: completed`。缺少 `poll.get("task_id") == task_id` 校验。

**P0 修复**：
```python
if poll.get("status") == "completed" and poll.get("task_id") == task_id:
    return {...}
```
同时 `research_async()`（第 182 行起）有完全相同 bug，要同步修。

### 🚨 Step 6：`research --add-sources`（F 级，P0）

**测试**：`research --topic "AI Agent 长期记忆..."`

**结果**：研究阶段同样 7s 完成（疑似又一次缓存命中），进入 import 阶段后：

```
⏳ [2/3] Importing 10 sources into notebook...
ERROR [notebooklm._core] RPC IMPORT_RESEARCH failed after 30.600s
Traceback (most recent call last): … 80 行 raw traceback 直接打到用户
notebooklm.exceptions.RPCTimeoutError: Request timed out calling IMPORT_RESEARCH
```

**问题**：
1. **P0 — traceback 泄漏到终端**：`cmd_research` 没有 try/except 包裹 `import_research_sources`，任何 RPC 错误都以 Python 栈回溯形式暴露给用户，对普通用户极不友好
2. **P0 — 30s 超时偏短**：10 个源的 import 默认 30s 易超时。应该：
   - 调大到 90-120s
   - 做分批（5 sources/batch）
   - 失败时返回 `{"status": "partial", "imported": N, "failed": M}` 而不是崩溃
3. **P1 — 默认 add-sources 应为 ON**：用户希望 research 默认写入笔记本（仅在 `--no-add-sources` 或 `--target-notebook` 时变更）。当前 `add_sources: false` 是默认，与 CLAUDE.md 中"用户 only"标记冲突，且违背用户预期。

## 使用感受总结

**好的部分**：
- `setup` 和 `ask` 是真正可用的功能，质量高于预期
- 回答的结构化程度和引用完整性是 NotebookLM 核心优势，体现得充分
- 降级建议（low confidence → suggest research）的用户引导设计不错

**糟糕的部分**：
- `plan` 和 `research` 在核心承诺上失败（打分、新鲜研究结果）
- 错误处理一致性差：有的优雅（ask 的 low confidence），有的灾难（research 的 traceback）
- 等待体验缺乏耐心感知：60s+ 任务没有动态进度，容易让用户误以为 hang 死

**V0.1 发布决策**：修完 4 个 P0 + 至少 P1 #5/#6 即可发布。`plan` 的 scoring 是重点战场。
